# Proposal 005 — API Sync Client

## Status: Draft

## Prerequisites
Proposal 004 (Local Storage & Offline Queue) — requires the `sync_queue` table and
`StorageService` interface for enqueue/dequeue and status transitions.

## Scope
- Tasks: ~3
- Layers: core/network, features/api_sync
- Risk: Medium — network reliability, retry correctness, background execution limits

---

## Problem Statement

Approved transcripts sit in the local `sync_queue` with status `pending` but have no
way to leave the device. The app needs an HTTP transport layer that reads from the queue,
POSTs each transcript to the user-configured API endpoint, handles transient and
permanent failures distinctly, and retries with backoff — all while respecting the
device's connectivity state. Without this, the app is a local-only recorder with no
integration value.

---

## Are We Solving the Right Problem?

**Root cause:** There is no outbound HTTP path from the sync queue to the user's API.
The queue exists (004) but nothing drains it.

**Alternatives dismissed:**
- *Firebase / cloud relay:* Adds a third-party intermediary the user did not ask for.
  The spec says "sends transcripts to user's API" — direct POST is the correct model.
- *Background isolate with `workmanager`:* Adds significant platform-specific complexity
  (Android JobScheduler, iOS BGTaskScheduler). For MVP the sync worker runs in the
  foreground Dart isolate while the app is open. Background sync is a follow-up.
- *WebSocket / persistent connection:* Over-engineered for infrequent, small payloads.
  Simple HTTP POST per transcript is sufficient.

**Smallest change?** Yes — this proposal adds only the HTTP client, sync worker loop,
and connectivity listener. Settings UI is deferred to 006; storage is already in 004.

---

## Goals

- Drain the sync queue by POSTing each pending transcript to the user's API endpoint
- Handle transient failures with exponential backoff and a bounded retry count
- Pause sync automatically when the device is offline and resume when connectivity returns

## Non-goals

- No batch sending (multiple transcripts per request) — single-item POST only for MVP
- No custom HTTP headers beyond `Authorization` and `Content-Type` for MVP
- No background sync when the app is suspended — foreground only
- No settings UI — that is Proposal 006
- No webhook signature verification or mutual TLS

---

## User-Visible Changes

After this proposal, when a transcript is approved and the device is online, it is
automatically sent to the configured API endpoint. If the device is offline or the
request fails transiently, the transcript stays queued and retries later. The user sees
sync status per transcript (pending / sent / failed) in the history list
(wired by Proposal 007). The `sending` state is transient (lasts only during
the HTTP request) and is not surfaced in the history UI.

### API Contract

The `ApiClient` sends one transcript per request.

**Request:**

```
POST {apiConfigProvider.url}
Authorization: Bearer {apiConfigProvider.token}
Content-Type: application/json

{
  "text": "transcript text here",
  "timestamp": 1710000000,
  "language": "pl",
  "deviceId": "uuid-of-device"
}
```

All fields are required. `timestamp` is Unix epoch seconds (int). `language` is the
ISO 639-1 code used during recognition. `deviceId` is the persistent device identifier
generated by Proposal 004.

**Response handling:**

| Status Code | Classification | Action |
|-------------|---------------|--------|
| 2xx | Success | Mark `sent`, stop retrying |
| 400, 401, 403, 404, 422 | Permanent failure | Mark `failed` with error message, do not retry |
| 408, 429, 5xx | Transient failure | Increment `attempts`, mark `failed`, schedule retry |
| Network error / timeout | Transient failure | Same as above |

The `ApiClient` returns a sealed result type:

```
ApiResult
  +-- ApiSuccess
  +-- ApiPermanentFailure { int statusCode, String message }
  +-- ApiTransientFailure { String reason }
```

**Dio configuration:**

| Setting | Value |
|---------|-------|
| Connect timeout | 10 seconds |
| Receive timeout | 15 seconds |
| Follow redirects | false |
| Max redirects | 0 |

### SyncWorker Behavior

The `SyncWorker` is a long-lived object managed by a Riverpod provider. It is started
when the app initializes (if auto-send is enabled) and stopped on dispose.

**Drain loop:**

0. Check `apiConfigProvider.url`. If null or empty, sleep 5 seconds, go to step 0.
1. Promote eligible retries: query `failed` items where `last_attempt_at + backoff < now`
   and `attempts < 10`, call `StorageService.markPendingForRetry(id)` for each.
2. Query `StorageService.getPendingItems()` — returns items ordered by `created_at` ASC (FIFO).
3. If the list is empty, sleep for 5 seconds, then go to step 0.
4. Pick the first item. Call `StorageService.markSending(id)`.
5. Load the full `Transcript` by `transcript_id`.
6. Call `ApiClient.post(transcript)`.
7. On `ApiSuccess`: call `StorageService.markSent(id)` (deletes sync_queue row). Go to step 0.
8. On `ApiPermanentFailure`: call `StorageService.markFailed(id, message)`. Go to step 0.
9. On `ApiTransientFailure`: if `attempts >= 10`, call `StorageService.markFailed(id, "max retries exceeded")` — permanent, no retry. Otherwise call `StorageService.markFailed(id, reason)`. Go to step 0.

**Retry transition:** Items with status `failed` and `attempts < 10` are retried.
The worker calls `StorageService.markPendingForRetry(id)` for eligible items before
each drain cycle. Eligibility is determined by comparing `last_attempt_at + backoff`
against the current time. Items whose backoff has not elapsed are skipped.

**Backoff schedule (per item, based on attempt count):**

| Attempt | Delay before next retry |
|---------|------------------------|
| 1 | 30 seconds |
| 2 | 1 minute |
| 3 | 5 minutes |
| 4 | 15 minutes |
| 5–9 | 1 hour |
| 10 | No retry — marked `failed` permanently |

**Backoff enforcement mechanism:** At the start of each drain cycle, the worker queries
`failed` items from `sync_queue` and checks `last_attempt_at + backoff_for_attempt_count`
against the current time. For eligible items (backoff elapsed, `attempts < 10`), it
calls `StorageService.markPendingForRetry(id)` to transition them back to `pending`.
Then it proceeds with step 1 (`getPendingItems()`), which returns only `pending` items.

This means the worker does not need its own timer per item — it simply polls, promotes
eligible failed items to pending, and processes the pending queue.

**Worker states:**

```
[idle] --start()--> [running] --pause()--> [paused] --resume()--> [running]
                       |                                             |
                       +--stop()-------> [stopped] <----stop()-------+
```

### Connectivity-Aware Behavior

`ConnectivityService` wraps `connectivity_plus` and exposes:

```
Stream<ConnectivityStatus> get statusStream;  // online | offline
Future<ConnectivityStatus> get currentStatus;
```

`ConnectivityStatus` is an enum: `{ online, offline }`. Any `ConnectivityResult` value
other than `none` maps to `online`.

The `SyncProvider` listens to `statusStream`:
- On transition to `offline`: calls `worker.pause()`.
- On transition to `online`: calls `worker.resume()` — the worker immediately re-enters
  the drain loop, flushing any items that accumulated while offline.

### API Configuration Provider (stub)

This proposal defines a stub `apiConfigProvider` that `ApiClient` reads from:

```
class ApiConfig {
  final String? url;      // null = not configured
  final String? token;    // null = no auth
}

final apiConfigProvider = Provider<ApiConfig>((ref) {
  return ApiConfig(url: null, token: null);  // stub — no URL configured
});
```

The stub defaults to `url: null`, which means the `SyncWorker` skips sending (no URL
to POST to). Proposal 006 (Settings Screen) replaces this stub with a real provider
that reads from `SharedPreferences` / `flutter_secure_storage`. This allows 005 to be
implemented and tested independently of 006.

The `SyncWorker` checks `apiConfigProvider.url` before each drain cycle. If null or
empty, it skips the cycle (no HTTP request attempted).

## Acceptance Criteria

1. `flutter analyze` exits with zero issues.
2. `flutter test` passes — all new tests green.
3. Given a `pending` sync item and a reachable mock server returning 200, the worker calls `markSent` (deleting the sync_queue row) within one drain cycle.
4. Given a mock server returning 500, the worker increments `attempts` and marks the item `failed`; the item becomes eligible for retry after the backoff delay.
5. Given a mock server returning 400, the worker marks the item `failed` permanently with no retry.
6. After 10 transient failures on the same item, the worker marks it `failed` with reason "max retries exceeded" and does not retry again.
7. When `ConnectivityService` emits `offline`, the worker pauses (no HTTP requests are attempted). When it emits `online`, the worker resumes and processes pending items.
8. The `ApiClient` sends the exact JSON shape specified in the API Contract section (fields: `text`, `timestamp`, `language`, `deviceId`).
9. Dio connect timeout is 10 s and receive timeout is 15 s.
10. This proposal defines a stub `apiConfigProvider` with `url: null`. The `SyncWorker` skips drain cycles when URL is null. Proposal 006 replaces the stub with a real provider.

## Risks

| Risk | Mitigation |
|------|------------|
| iOS suspends the app and the sync worker stops mid-queue | Worker is foreground-only for MVP. Document that the app must be open for sync to run. Background sync is a post-MVP follow-up using `workmanager`. |
| User's API is slow, causing timeouts that fill the retry queue | Dio timeouts are bounded (10 s connect, 15 s receive). Retry cap of 10 prevents infinite growth. Items are marked permanently failed after cap. |
| `connectivity_plus` reports `online` but the network is unreliable (e.g., captive portal) | The worker treats network errors the same as `offline` — it retries with backoff. Connectivity status is a hint to pause, not a guarantee of reachability. |
| Race condition if user approves transcripts faster than worker drains | FIFO ordering and single-item processing prevent out-of-order sends. Queue depth may grow temporarily but drains once the API catches up. |

## Known Compromises and Follow-Up Direction

### No batch sending (MVP pragmatism)
Each transcript is sent as a separate HTTP request. If users produce many transcripts
in quick succession, this is chatty. A future proposal can add batch mode
(`POST` with an array body) behind a feature flag.

### No custom headers beyond Authorization
The MVP sends only `Authorization: Bearer {token}` and `Content-Type: application/json`.
If users need custom headers (e.g., `X-Api-Key`, `X-Workspace-Id`), add a key-value
header editor in settings as a follow-up.

### Foreground-only sync
The worker only runs while the app is in the foreground. For true background sync,
a follow-up proposal should integrate `workmanager` (Android) and `BGTaskScheduler`
(iOS) with appropriate platform configuration.

### Backoff is poll-based, not timer-based
The worker polls every 5 seconds and skips items still in backoff. This is simple but
means an item with a 1-hour backoff is checked (and skipped) many times before becoming
eligible. At MVP scale this is negligible. A timer-based approach can replace it if
polling overhead becomes measurable.
