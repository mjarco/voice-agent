# Proposal 006 — Settings Screen

## Status: Implemented

## Prerequisites
- Proposal 004 (Local Storage) — provides device ID via `StorageService.getDeviceId()`, displayed in the About section.
- Proposal 005 (API Sync Client) — provides the stub `apiConfigProvider` that this proposal replaces with a real settings-backed provider. Also provides `ApiClient` used by Test Connection.
- Proposal 008 (App Navigation) — provides the `/settings` tab route with a placeholder screen that this proposal replaces. Also provides the stub `apiUrlConfiguredProvider` that this proposal replaces.

## Scope
- Tasks: ~4
- Layers: features/settings, core/network (reads config)
- Risk: Low — standard form UI with persistence, no complex logic

---

## Problem Statement

The API sync client (005) requires a URL and an optional auth token, but there is no
UI for the user to enter or change these values. Other preferences (language, auto-send,
history retention) also need a home. Without a settings screen, configuration would
have to be hardcoded or passed through debug tooling, making the app unusable for
anyone other than the developer.

---

## Are We Solving the Right Problem?

**Root cause:** There is no user-facing surface to view or edit application configuration.

**Alternatives dismissed:**
- *First-run wizard / onboarding flow:* Adds a separate navigation flow and
  "completed onboarding" state. A single settings screen that is always accessible is
  simpler and covers the same inputs. Onboarding can wrap it later if needed.
- *Environment variables / .env file:* Not user-accessible on a mobile device. Settings
  must be editable at runtime.
- *Store everything in SharedPreferences (including the token):* The API token is a
  secret. `SharedPreferences` stores data in plaintext XML (Android) or plist (iOS).
  `flutter_secure_storage` uses the platform keychain/keystore and is the correct
  choice for secrets.

**Smallest change?** Yes — this proposal adds only the settings UI, persistence layer,
and a test-connection action. It does not modify the sync worker or API client
internals; it only provides the configuration values they already read from providers.

---

## Goals

- Give the user a single screen to configure API endpoint, auth token, and app preferences
- Persist the API token securely using platform-native encrypted storage
- Let the user verify their API configuration with a "Test Connection" action before relying on it

## Non-goals

- No onboarding flow or setup wizard
- No per-profile or multi-account settings
- No export/import of settings
- No advanced HTTP configuration (custom headers, client certificates)

---

## User-Visible Changes

A new "Settings" screen accessible from the bottom navigation bar (wired by 008).
The screen contains grouped form fields for API configuration, transcription preferences,
and general preferences. The user can enter an API URL and token, tap "Test Connection"
to verify, and toggle auto-send and history retention. All changes persist immediately
on edit (no save button).

## Solution Design

### Directory Structure

```

      settings_screen.dart       # UI — grouped list of settings

      settings_provider.dart     # Riverpod providers exposing settings state

test/
  features/
    settings/
      settings_service_test.dart
      settings_provider_test.dart
      settings_screen_test.dart
```

### Settings Model

```
AppSettings {
  // API configuration
  String? apiUrl
  String? apiToken          // stored separately in secure storage
  bool autoSend             // default: true

  // Transcription
  String language            // default: "auto" — values: "auto", "pl", "en"

  // General
  bool keepHistory           // default: true

  // Read-only (not user-editable)
  String deviceId            // generated by 004, displayed for reference
  String appVersion          // from package_info
}
```

### Storage Split

| Setting | Storage | Key | Rationale |
|---------|---------|-----|-----------|
| API URL | SharedPreferences | `api_url` | Not a secret |
| API Token | flutter_secure_storage | `api_token` | Secret — must be encrypted at rest |
| Auto-send | SharedPreferences | `auto_send` | Not a secret |
| Language | SharedPreferences | `language` | Not a secret |
| Keep history | SharedPreferences | `keep_history` | Not a secret |
| Device ID | SharedPreferences | `device_id` | Generated by 004, read-only here |

### Settings Screen Layout

The screen uses a `ListView` with `ListTile`-based rows grouped into sections.
No custom widgets — standard Material 3 components throughout.

```
Settings
-----------------------------------------------
API Configuration
  API URL              [TextField — https://...]
  API Token            [TextField — obscured, ••••••••]
  Test Connection      [OutlinedButton]
                       Status indicator: idle / testing / success / error

Transcription
  Language             [DropdownButton — Auto / Polish / English]

General
  Auto-send            [SwitchListTile — ON by default]
  Keep history         [SwitchListTile — ON by default]

About
  Version              1.0.0 (non-editable)
  Device ID            uuid (tap to copy to clipboard)
-----------------------------------------------
```

**Field behaviors:**

- **API URL:** Validated on focus loss. Must start with `http://` or `https://`. If
  invalid, show inline error "Enter a valid URL starting with http:// or https://".
- **API Token:** Obscured by default. Optional — the field may be left empty if the
  user's API does not require auth. Saved to secure storage on focus loss.
- **Test Connection button:** Disabled when API URL is empty or invalid. Enabled
  otherwise.
- **Language dropdown:** Values map to ISO 639-1 codes: `auto` (let the STT engine
  detect), `pl`, `en`. Saved immediately on change.
- **Auto-send / Keep history toggles:** Saved immediately on change.
- **Device ID:** Tapping copies to clipboard and shows a `SnackBar` "Device ID copied".

### Test Connection Behavior

When the user taps "Test Connection":

1. Validate the API URL format. If invalid, show inline error and abort.
2. Show a loading indicator on the button (circular progress).
3. Send an HTTP POST to the configured URL with:
   ```
   Authorization: Bearer {token}   // omitted if token is empty
   Content-Type: application/json

   { "test": true }
   ```
4. Wait for response (uses the same Dio timeouts as `ApiClient`: 10 s connect, 15 s receive).
5. Display result:
   - **2xx response:** Green check icon + "Connection successful"
   - **4xx response:** Orange warning icon + "Server returned {status code}"
   - **5xx response:** Red error icon + "Server error ({status code})"
   - **Network error / timeout:** Red error icon + "Could not reach server"
6. Result indicator remains visible until the user edits the URL or token.

The test connection uses `ApiClient.post()` from 005 directly — no separate HTTP logic.

### SettingsService Interface

```
abstract class SettingsService {
  Future<AppSettings> load();
  Future<void> saveApiUrl(String url);
  Future<void> saveApiToken(String token);    // writes to secure storage
  Future<void> saveAutoSend(bool value);
  Future<void> saveLanguage(String language);
  Future<void> saveKeepHistory(bool value);
  Future<String?> readApiToken();             // reads from secure storage
}
```

Each setter writes a single key immediately (no batch save). `load()` reads all keys
from both `SharedPreferences` and `flutter_secure_storage` and returns a complete
`AppSettings` instance.

### Riverpod Integration

```
settingsServiceProvider      → Provider<SettingsService>
appSettingsProvider          → StateNotifierProvider<SettingsNotifier, AppSettings>
```

`SettingsNotifier` loads settings on initialization and exposes methods that delegate
to `SettingsService` and update state.

**Stub replacement:** Proposal 005 defines a stub `apiConfigProvider` (returns null URL).
This proposal replaces that stub: `appSettingsProvider` becomes the source of truth for
API URL and token, and `apiConfigProvider` is overridden to read from `appSettingsProvider`.
Similarly, Proposal 008's stub `apiUrlConfiguredProvider` is replaced to read the real
URL from settings.

## Acceptance Criteria

1. `flutter analyze` exits with zero issues.
2. `flutter test` passes — all new tests green.
3. The settings screen renders with all sections (API Configuration, Transcription, General, About).
4. Entering a valid URL and tapping "Test Connection" against a reachable mock server showing 200 displays "Connection successful".
5. Entering an invalid URL (e.g., "not-a-url") and leaving the field shows inline validation error.
6. The API token is stored in `flutter_secure_storage`, not in `SharedPreferences` — verifiable by checking that `SharedPreferences` contains no `api_token` key.
7. Toggling "Auto-send" off and reopening the app preserves the off state.
8. Selecting a language in the dropdown persists immediately — no save button needed.
9. Tapping the Device ID row copies the value to the clipboard and shows a SnackBar.
10. "Test Connection" button is disabled when the API URL field is empty.

## Risks

| Risk | Mitigation |
|------|------------|
| `flutter_secure_storage` requires Android minSdk 23+ for KeyStore | Project minSdk is 24 (set in 000), which satisfies the requirement. |
| Some Android devices have broken KeyStore implementations causing `flutter_secure_storage` to throw | Wrap secure storage reads in try-catch. If read fails, treat token as absent and prompt user to re-enter. Log the error for diagnostics. |
| User enters a URL that is valid in format but points to a non-existent server | "Test Connection" provides immediate feedback. The sync worker (005) also handles unreachable servers gracefully via retry. |
| SharedPreferences is cleared if user clears app data | Expected platform behavior. Token in secure storage may also be cleared. User will need to re-enter settings. No mitigation needed — this is standard. |

## Known Compromises and Follow-Up Direction

### No onboarding flow (MVP pragmatism)
New users must navigate to settings manually and enter their API URL before sync works.
A first-run onboarding flow that walks the user through configuration can be added
as a follow-up proposal once the settings infrastructure exists.

### No settings migration
Settings are stored as flat key-value pairs with no versioning. If key names or
semantics change in a future proposal, a migration step will be needed. For MVP the
schema is simple enough that this is not a concern.

### Auto-save on every field change
There is no explicit "Save" button — each field persists immediately. This is simple
and removes the risk of lost changes, but means partial edits (e.g., typing a URL
character by character) trigger writes. SharedPreferences writes are cheap enough
that this is acceptable. For the token field, saving on focus loss avoids character-
by-character writes to secure storage.
